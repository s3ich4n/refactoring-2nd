# Preface

- 리팩터링은 소프트웨어가 "썩지 않게" 하는 방어기제다.
- 리팩터링은 동작은 그대로인데 구조를 바꾸는 것이다.
    - 소프트웨어니까 내부 설계를 개선할 수 있다.
- 그런데 리팩터링은 체계적이고 계획적으로 해야한다.
- 테스트코드가 없는 리팩터링은 효용이 없거나 매우 떨어진다.
- 그렇다고 마냥 어려운 건 아니다.
    - 한 클래스의 필드를 다른 클래스로 옮기거나
    - 일부 코드를 메소드 밖으로 빼거나
    - 코드 일부를 상속구조의 위/아래로 옮기기만 해도
    - ...이게 쌓이면 설계가 개선될 수 있다.
